1. Цель доработки
Полностью переработать UI-слой текущей складской системы. Итоговый интерфейс должен выглядеть как современный продукт, а не как служебная форма. Система обязана оставаться строгой, быстрой и пригодной для интенсивной складской работы, но при этом получить нормальную визуальную иерархию, читаемые экраны, карточную аналитику, понятные статусы, чёткие действия и мобильные сценарии для объекта.
2. Анализ текущего UI и обязательные проблемы к исправлению
• Слишком пустые экраны: полезное пространство есть, но информационная плотность низкая, пользователь не получает быструю картину по складу.
• Слабая иерархия: заголовки, поля, таблицы и кнопки имеют одинаковую визуальную силу, из-за чего пользователь долго ищет главное действие.
• Формы прихода, выдачи и перемещения выглядят как технические бланки без сценарной логики: нет пошагового маршрута, подсказок, статусов заполнения.
• Навигация слева функционально правильная, но слишком монотонная; разделы не группируются по смыслу и не дают ощущения модульной системы.
• Нет карточек KPI, быстрых действий, overdue-блоков, проблемных задач и явных индикаторов риска.
• Нет нормальной карточки сущности: выдача, наряд, объект, инструмент, документ должны открываться в detail-view или side-panel, а не только в таблице.
• Таблицы без сильных фильтров и без цветовой логики статусов. Нет сохранённых фильтров, массовых действий и колонок с фиксированными приоритетами.
• Документный слой слишком слабый: нужна структура по вкладкам, превью, метки типов документов, история версий, привязка к сущности и drag-and-drop загрузка.
• Слабая работа с инструментом: QR есть только как функция, но не как отдельный сценарий учёта, поиска, возврата, ремонта и спорных случаев.
• Мобильный сценарий фактически не описан: для объекта нужны быстрые экраны выдачи, подтверждения, сканирования QR и фотофиксации.
3. Целевое визуальное направление
• Стиль: строгий enterprise UI для складской системы строительной компании.
• Характер: чистый, плотный, уверенный, без декоративного мусора.
• Палитра: глубокий тёмно-синий sidebar, светлый контентный фон, белые карточки, акцентный синий для primary-действий, зелёный/жёлтый/красный для статусов.
• Компоненты: скругления 10–14 px, мягкие тени, чёткие отступы, контрастные заголовки, не перегруженные формы.
• Интерфейс должен собираться из единой дизайн-системы: tokens, grid, spacing, colors, buttons, inputs, tabs, drawers, tables, badges, cards и alerts.

5. Экран «Главная» — обязательная новая структура
• Верхняя зона: breadcrumbs, глобальный поиск, центр уведомлений, пользовательское меню, быстрый переход к QR-сканированию.
• Первый ряд карточек KPI: поступления, выдачи, перемещения, лимиты, на согласовании, просроченные задачи. Каждая карточка кликабельна и ведёт в отфильтрованный список.
• Левый нижний блок: «Остатки по складам» с donut / stacked chart и легендой по складам.
• Центральный блок: таблица top-materials с колонками Материал / Ед. / Остаток / Резерв / Доступно / Риск дефицита.
• Правый блок: «Проблемы сегодня» и «Быстрые действия» (новое поступление, новая выдача, новое перемещение, новый наряд, заявка на материалы).
• Нижние блоки: последние поступления, открытые выдачи на бригадиров, неподтверждённые возвраты, документы без привязки.
• Dashboard должен быть настраиваемым: пользователь выбирает порядок и видимость блоков.
6. Формы операций: новый UX-подход
Любая форма прихода, выдачи и перемещения должна перестать быть «листом полей». Новая форма строится как сценарий из четырёх зон:
• Шапка документа: номер, статус, дата, объект, склад, ответственный, основание.
• Основной блок позиций: табличный ввод строк с автопоиском, подсказками, остатками, лимитами, QR, вложениями и комментариями.
• Правая контекстная панель: краткая сводка, лимит/факт, предупреждения, связанные документы, история действий.
• Нижняя action-zone: Черновик / Проверить / Провести / Отправить на согласование / Печать / Возврат / Отмена.

8. Новые функции, которые нужно добавить в UI-слой
• Global search поверх всей системы: материалы, инструменты, документы, объекты, выдачи, наряды, пользователи.
• Universal side drawer / detail panel для любой сущности.
• Saved filters и shared views для кладовщиков, РП, руководства.
• Проблемные статусы и overdue-бейджи с цветовой логикой.
• Таймлайн событий по документу и по инструменту.
• QR scan mode в mobile: выдача, поиск, возврат, проверка принадлежности.
• Quick actions panel на dashboard и внутри списков.
• Массовые действия над таблицами: подтверждение, экспорт, смена статуса, назначение ответственного.
• Пустые состояния экранов с понятными CTA, а не пустой серый холст.
• Inline attachments: фото, PDF, УПД, ТН, паспорта материала, акты, комментарии.
9. Мобильная версия / PWA — жёсткие сценарии
• Главная: KPI + просрочки + открытые выдачи + кнопка сканирования QR.
• Выдачи: табы «Открытые / Мои / Все», карточки с цветовым статусом, быстрый вход в деталь.
• QR-сканер: полноэкранный режим, подсветка рамки, сценарии выдать / найти / вернуть / проверить.
• Материалы: мобильный каталог с поиском, остатками, доступностью, переходом в detail.
• Подтверждение выдачи и фотофиксация должны занимать не более 2–3 экранов.
• Все кнопки — крупные, нижняя навигация — 4–5 ключевых разделов, critical CTA закреплены снизу.

11. Жёсткие требования к таблицам
• Таблицы — это основной рабочий слой системы. Они должны быть быстрыми, не перегруженными, но мощными.
• Обязательны: сортировка, фильтры, фиксированные колонки, видимость колонок, поиск по строкам, массовое выделение.
• Первая колонка — ключевой идентификатор / номер документа, вторая — главная сущность, далее контекст, статус, дата, действия.
• В строке должны быть быстрые действия: открыть, провести, отправить на согласование, печать, меню ещё.
• Для длинных списков обязательны skeleton/loading-state, empty-state и сохранение пользовательской конфигурации колонок.

13. UI для инструментов и QR — отдельный обязательный контур
• Добавить модуль «Инструмент» в левую навигацию.
• Карточка инструмента: фото, название, инвентарный номер, QR, серийный номер, статус, объект, склад, бригадир, история.
• QR генерируется прямо из карточки и печатается на наклейку.
• После сканирования пользователю показывается крупная карточка инструмента с безопасными действиями: выдать / вернуть / открыть детали / зафиксировать поломку.
• Обязательны статусы ремонта, потери, повреждения, списания и спорного случая.

15. Что нужно сделать Cursor AI
• Сначала собрать design system: colors, typography, buttons, forms, badges, cards, tables, drawers, tabs, alerts, uploader, skeletons.
• Затем собрать layout shell: sidebar, topbar, breadcrumbs, notification center, user menu, global search.
• Дальше собрать новые desktop screens по приоритету: Главная → Выдачи → Поступления → Перемещения → Материалы → Инструмент → Наряды → Лимиты → Документы.
• Затем собрать mobile/PWA screens: Главная, Выдачи, QR Scan, Материалы, Подтверждение.
• Каждый экран должен иметь состояния loading / empty / data / error / success / partial / permission denied.
• Каждый экран должен иметь mock data fixtures и потом переключаться на реальные API.
16. Definition of Done по UI-слою
• Нет ни одного «пустого серого» экрана без смысла и CTA.
• Главная страница даёт управленческую картину за 5–10 секунд.
• Любая операция открывается либо в detail view, либо в drawer, либо в полноценной карточке.
• Формы операций читаются как сценарий, а не как набор сырых полей.
• Статусы, риски, просрочки и согласования видны без лишних кликов.
• QR-сценарий инструмента работает и на desktop, и на mobile.
• Мобильная версия пригодна для реальной работы на объекте.
• UI выглядит как единый продукт, а не как набор случайных страниц.

18. Финальная формулировка для команды
Текущий UI допускается использовать только как источник структуры модулей. Визуально и сценарно он должен быть полностью переработан. Итоговая система обязана получить современный enterprise-интерфейс с сильной dashboard-логикой, detail-панелями, продуманными формами операций, карточками сущностей, нормальной работой со статусами, QR-сценариями, документами и мобильными экранами для объекта. Любая реализация «как сейчас, только аккуратнее» считается неприемлемой.
